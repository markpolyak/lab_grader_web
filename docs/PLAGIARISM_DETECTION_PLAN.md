# План разработки: автоматическая проверка на плагиат

Черновик исследования и план реализации фичи автоматической проверки студенческих работ на плагиат.

## 1. Постановка задачи

Сейчас (июль 2026) проверка на плагиат в проекте выполняется вручную через **MOSS** (Measure Of Software Similarity, Stanford University):

- Конфигурация уже описана в YAML-курсах (секция `moss:` в `courses/*.yaml`, задокументирована в `docs/COURSE_CONFIG.md`), но **в коде она нигде не используется** — ни `main.py`, ни модули `grading/*.py` не содержат обращений к MOSS. Проверка осуществляется с помощью кода из прошлой версии этого проекта: https://github.com/markpolyak/lab-grader/blob/dev1/main.py (функция `check_plagiarism`)
- MOSS требует выгрузки **всего корпуса** работ единым запросом на сервера Stanford, поэтому реально проверка проводится вручную в конце семестра, когда все студенты уже сдали работы.

Желаемое поведение: проверка на плагиат должна выполняться **в момент простановки оценки** (когда работа проходит CI и заносится в Google Таблицу как `v`), а не пост-фактум по всему курсу. В идеале — без зависимости от внешнего сервиса, локально.

## 2. Возможно ли проверять в момент сдачи?

Да, но не с MOSS в текущей модели использования. MOSS проектировался как batch-сервис: чтобы получить осмысленные попарные совпадения, нужно каждый раз отправлять весь корпус (или хотя бы всё накопленное на текущий момент) на сервера Stanford. Постоянная пересылка растущего корпуса на каждую галочку неэффективна и упирается в очередь/квоты сервиса.

Локальные инструменты этой проблемы не имеют: система сама хранит локальный кэш уже проверенных работ и при каждой новой сдаче сравнивает **только новую работу против уже накопленного корпуса**, добавляя её в кэш по итогу. Это даёт «проверку в момент сдачи» без внешней зависимости и без пересчёта всего корпуса каждый раз.

## 3. Сравнение инструментов

| Критерий | MOSS (текущий) | JPlag | Dolos | compare50 |
|---|---|---|---|---|
| Тип | внешний сервис Stanford | локальный, Java | локальный, Node.js | локальный, Python |
| Лицензия/поддержка | закрытая, бесплатно для вузов | open source, активно развивается (KIT) | open source, активно развивается (Ghent University / Dodona) | open source, активно развивается (Harvard CS50, релизы в 2026) |
| Языки | много (через `-l`) | 20+: Java, C/C++, C#, Python, JS/TS, Go, Kotlin, Rust, Scala и др. | много через tree-sitter: C/C++/C#, Python, JS/TS, Java, PHP, Bash и др. | 300+ языков (более общий текстово-структурный подход) |
| Устойчивость к обфускации/перестановкам блоков | средняя | **лучшая** — отдельно исследовалась в академических работах | средняя (reorder — рабочая атака) | средняя (набор алгоритмов: structure/text/exact-winnowing) |
| Рантайм в Docker (сейчас `python:3.12-slim`) | не нужен (нужна только сеть) | нужен JRE (+100–200 МБ к образу) | нужен Node.js + нативные tree-sitter биндинги (сложнее multi-arch сборка) | **`pip install compare50` — без нового рантайма** |
| "Инкрементальность" (новая работа vs накопленный корпус) | нет, только batch над всем корпусом | да — запуск инструмента над локальной директорией-кэшем | да, аналогично | да, аналогично |
| Готовый отчёт для преподавателя | веб-страница на серверах Stanford | локальный HTML-отчёт (JPlag Report Viewer) | интерактивный веб-отчёт (`-f web`), можно захостить | текст/HTML вывод, менее наглядно "из коробки" |
| Исключение шаблонного/базового кода | `moss.basefiles` (уже в конфиге) | Base Code (`-bc`) — та же идея | поддержка ignore/template-файлов | exclude-паттерны |
| Кросс-семестровые сравнения (`moss.additional`) | поддерживается | поддерживается (старые репо просто добавляются в корпус) | поддерживается | поддерживается |

Общая оговорка для всех перечисленных инструментов: ни один не предназначен для детекции ИИ-сгенерированного кода — это отдельная задача, не входящая в объём данной фичи.

Существуют также коммерческие API (например, Codequiry, позиционируется как "MOSS alternative"), но они противоречат цели "не зависеть от внешних сервисов" и рассматриваются только как запасной вариант, а не основной путь.

### Рекомендация

Начать с **`compare50`** — единственный вариант, который встраивается как обычная pip-зависимость без нового рантайма в Docker-образе (сейчас чистый Python-стек `python:3.12-slim`). Минимальный риск: не нужно менять `backend.Dockerfile` под JRE/Node.js.

Если на этапе предварительного исследования (см. план ниже) окажется, что устойчивости к обфускации/перестановкам не хватает — что особенно критично для C/C++ лабораторных, где студенты чаще переставляют блоки кода, — следующий кандидат **JPlag**: это специально изученная его сильная сторона, ценой добавления JRE в образ (умеренная сложность).

Dolos держим как третий вариант: самый красивый интерактивный веб-отчёт из коробки, но сборка нативных tree-sitter биндингов под multi-arch Docker (проект собирает образы в `.github/workflows/ci.yaml`) добавляет эксплуатационных сложностей, избыточных для быстрой реализации прототипа работающей фичи.

## 4. Предлагаемая архитектура

### 4.1 Точка встраивания

`main.py:grade_lab` (эндпоинт `POST /courses/{course_id}/groups/{group_id}/labs/{lab_id}/grade`) — после успешной записи `v...` в Google Таблицу, не блокируя ответ студенту. Используем встроенный в FastAPI `BackgroundTasks` — доп. инфраструктура (очереди, воркеры) не нужна на первом этапе.

### 4.2 Локальный кэш исходников

Скачиваются **только файлы, указанные в конфиге лабы** (`files:` / `moss.local-path`), а не репозиторий целиком — так же, как это уже было устроено в старой реализации на MOSS ([`check_plagiarism` в lab-grader](https://github.com/markpolyak/lab-grader/blob/dev1/main.py#L404)). Оттуда стоит переиспользовать по крайней мере два решения:

- **Структура каталога кэша**: `{local_path}/{org}/{repo_name}/{filename}` (в старой реализации — `local_path/org/repo/filename` через `os.path.join(local_path, *repo.split('/'))`). Для новой версии предлагается `plagiarism_cache/{course_id}/{lab_id}/{org}/{repo_name}/{filename}`.
- **Конвертация `.ipynb` → `.py`** перед сравнением (актуально для курсов по ML/анализу данных, где `files:` могут содержать ноутбуки) — в старом коде это `convert_ipynb_to_py()` через `nbconvert.PythonExporter`. Инструменты сравнения (compare50/JPlag) работают с исходным кодом, а не с JSON-структурой ноутбука, поэтому конвертация обязательна.

Скачивание — через GitHub API (можно переиспользовать/расширить `grading/github_client.py`, там уже есть похожий метод получения содержимого файла).

**Хранение — постоянное** (не ротируется), в т.ч. чтобы поддерживать кросс-семестровые сравнения (`additional`) без повторного скачивания старых работ при каждом запуске. По духу аналогично уже существующему volume `logs/` — та же директория-том в Docker, тот же принцип бэкапа (или его отсутствия) на усмотрение владельца инфраструктуры.

### 4.3 Сравнение

Новый модуль `grading/plagiarism.py`:
- вызывает выбранный движок subprocess'ом над директорией кэша конкретной лабы;
- парсит результат в нормализованный список `(student_a, student_b, similarity, details)`;
- поддерживает исключение базового/шаблонного кода (`basefiles`) и кросс-семестровые сравнения (`additional`), пред-заполняя кэш старыми репозиториями один раз (не при каждой сдаче).

### 4.4 Хранение результатов

Формат хранения: **SQLite** (модуль `sqlite3` из стандартной библиотеки Python, без новых зависимостей). Файл `plagiarism.db` хранится в том же volume, что и `plagiarism_cache/`.

Ориентировочная схема — одна таблица `plagiarism_matches`:

| Поле | Тип | Описание |
|---|---|---|
| `course_id` | text | ID курса |
| `lab_id` | text | ID лабораторной работы |
| `student_a`, `student_b` | text | GitHub-логины сравниваемой пары |
| `similarity` | real | доля совпадения, 0..1 |
| `details` | text | путь к детальному отчёту движка / сериализованные детали совпадения |
| `checked_at` | timestamp | момент проверки |
| `reviewed_by_teacher` | boolean | флаг ручного рассмотрения преподавателем (чтобы не флагировать повторно при пересчётах) |

Точная схема уточняется на этапе реализации (фаза 3, раздел 5).

### 4.5 Флаги для преподавателя

Основной канал — **комментарий (comment) к ячейке** лабораторной работы в Google Таблице, добавляется автоматически только если similarity выше порога. Сам грейд (`v`, `v@10.5` и т.д.) не изменяется.

Комментарии (в отличие от примечаний/notes) поддерживают тред-структуру: бот оставляет комментарий с информацией о проценте совпадения, преподаватель отвечает в этом же треде, подтверждая или опровергая плагиат.

Техническая реализация — Google Drive API v3, ресурс `comments` (`comments.create`, `comments.list`, `replies.create`, `replies.list`), не `gspread`-примечания (`insert_note`/`update_note` работают с другим механизмом — notes, не comments). Существующий сервис-аккаунт уже имеет scope `https://www.googleapis.com/auth/drive` (main.py), дополнительных разрешений не требуется.

Технические риски, требующие проверки на этапе предварительного исследования (раздел 5, фаза 0):
- формат поля `anchor` для привязки комментария к конкретной ячейке Google Sheets официально не задокументирован Google (документирован только для Google Docs); нужно проверить на практике, что комментарий действительно прикрепляется к нужной ячейке лабораторной работы, а не к файлу в целом;
- видимость комментариев для роли Viewer (студенты с доступом только на чтение) — нужно подтвердить эмпирически на реальной таблице курса.
- определение факта ответа/разрешения треда преподавателем (`replies.list` / статус `resolved`) для последующей синхронизации с полем `reviewed_by_teacher` (раздел 4.4).

Дополнительный канал — отдельная страница в админ-панели (`frontend/courses-front/src/components/`), обращающаяся к новому эндпоинту, например `GET /courses/{course_id}/labs/{lab_id}/plagiarism`, со списком пар выше порога и деталями совпадения.

### 4.6 Конфигурация курса

Обобщить секцию `moss:` до `plagiarism:`, сохранив `moss:` как deprecated-алиас для обратной совместимости с уже существующими 8 файлами курсов (`courses/*.yaml`), которые используют `moss:`.

Порог сходства (`threshold`) задаётся для каждой лабы, необязателен — если не указан, используется значение по умолчанию (константа в коде, например `DEFAULT_PLAGIARISM_THRESHOLD`).

```yaml
plagiarism:
  engine: compare50        # compare50 | jplag | dolos | moss (legacy alias)
  language: python          # используемый движком идентификатор языка
  threshold: 0.6            # необязательно; 0..1, порог для флага; если не задан — используется значение по умолчанию
  local-path: lab3          # поддиректория в репозитории с проверяемыми файлами
  additional:                # доп. GitHub-организации для сравнения (например, прошлые годы)
    - suai-os-2024
  basefiles:                 # шаблонный/базовый код, исключаемый из сравнения
    - repo: teacher/templates
      filename: lab3.cpp
```

После реализации нужно обновить `docs/COURSE_CONFIG.md` (заменить/дополнить раздел "MOSS (Проверка плагиата)") и `docs/PROJECT_DESCRIPTION.md`.

## 6. Открытые технические риски (проверить на этапе предварительного исследования)

- Формат поля `anchor` для привязки комментария Google Drive API v3 к конкретной ячейке Google Sheets — официально не задокументирован для Sheets (задокументирован только для Docs).

## 7. Не входит в объём данной фичи

- Детекция кода, сгенерированного ИИ (LLM) — ни один из рассмотренных инструментов для этого не предназначен, это отдельная задача.
- Полная замена ручного разбора преподавателем — фича только **флагирует** подозрительные пары для последующего ручного review, она не должна автоматически проставлять `x` вместо `v` по решению алгоритма similarity.
