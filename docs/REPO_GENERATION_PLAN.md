# План разработки: автоматическое создание репозитория студента (замена GitHub Classroom)

Требования и план реализации фичи.

## 1. Постановка задачи

GitHub Classroom прекращает работу. Из всего функционала GHC для проекта используется только один сценарий: преподаватель даёт студентам ссылку на задание, студент переходит по ней, и для него автоматически создаётся приватный репозиторий на основе репозитория-шаблона с правами доступа только у этого студента (и преподавателя/организации).

Дополнительно преподаватель сейчас поддерживает отдельный сторонний сервис **github-reinvite**, компенсирующий баг/особенность GitHub Classroom и GitHub API: приглашение в коллабораторы репозитория иногда не доходит до студента (письмо в спаме, уведомление не замечено), а через 7 дней GitHub автоматически аннулирует неподтверждённое приглашение — в результате репозиторий создан, но у студента нет доступа. Функционал этого сервиса нужно объединить с новой фичей создания репозитория, чтобы не поддерживать два отдельных инструмента.

Существующий процесс проверки лабораторных работ (`grade_lab`, main.py:597+), который по конвенции ожидает репозиторий с именем `{github-prefix}-{username}` в организации курса (main.py:632-633), и существующая регистрация студента (`/register`, main.py — связывание ФИО и GitHub username в Google Таблице) этой фичей **не затрагиваются** и должны продолжать работать без изменений.

## 2. Пользовательский сценарий

1. Преподаватель один раз для каждой лабы получает (копирует) ссылку вида `https://<host>/join/{course_id}/{lab_id}` и рассылает её студентам (чат, LMS и т.п.). Ссылка не одноразовая и не персональная — одна и та же для всех студентов курса на конкретную лабу.
2. Студент открывает ссылку, видит название курса и лабы и кнопку «Войти через GitHub и создать репозиторий».
3. Студент авторизуется через GitHub OAuth (если ещё не авторизован — подтверждает доступ; если уже авторизован ранее — шаг проходит без дополнительных кликов).
4. Система автоматически:
   - создаёт для студента приватный репозиторий `{github-prefix}-{username}` из репозитория-шаблона лабы (если такого репозитория ещё нет);
   - проверяет и при необходимости чинит доступ студента к этому репозиторию (см. §5).
5. Студент попадает на страницу успеха с прямой ссылкой на свой репозиторий на GitHub.

Ссылку можно посещать повторно в любой момент — например, если приглашение истекло через 7 дней, повторный переход по той же ссылке восстановит доступ (замена ручного запуска github-reinvite).

## 3. Аутентификация студента: разбор OAuth-механики

Используется **GitHub OAuth Web Application Flow**, а не ручной ввод логина в форму:
логин, введённый вручную, не верифицируется и не годится для действия с побочным эффектом (выдача доступа к приватному репозиторию постороннему аккаунту при опечатке). Ниже — подробный разбор механики, так как это самая нетривиальная часть фичи.

### 3.1 Действующие лица и настройка

- OAuth App регистрируется **преподавателем** на github.com (Settings → Developer settings → OAuth Apps), не студентом — это важно, см. обсуждение рисков ниже. При регистрации GitHub выдаёт:
  - `Client ID` — не секретный идентификатор, может быть виден где угодно, в том числе во frontend;
  - `Client Secret` — секретный, используется только backend'ом. Во frontend (Vite) любая переменная окружения с префиксом `VITE_` автоматически попадает в клиентский JS-бандл, поэтому `GITHUB_OAUTH_CLIENT_SECRET` не должен иметь такой префикс и не должен читаться нигде во frontend-коде.
  - **Authorization callback URL** — точный адрес, на который GitHub перенаправит браузер после подтверждения доступа (например, `https://<host>/join/callback`). GitHub сверяет этот адрес при каждом запросе и не отдаст код на любой другой URL — это встроенная защита от подмены получателя.
- Запрашиваемый scope — только `read:user` (идентификация). Права на репозитории токену студента не нужны — создание репозитория и управление доступом выполняется от имени серверного `GITHUB_TOKEN`, а не токена, полученного от студента.

### 3.2 Пошаговый разбор запросов

1. Браузер студента открывает `GET /join/{course_id}/{lab_id}/start` на нашем backend.
2. Backend формирует `state` (см. §3.3) и отдаёт редирект на:
   ```
   https://github.com/login/oauth/authorize
     ?client_id={GITHUB_OAUTH_CLIENT_ID}
     &redirect_uri={callback URL}
     &scope=read:user
     &state={подписанный state}
   ```
3. GitHub показывает студенту страницу подтверждения доступа (при первом обращении к этому OAuth App) либо сразу переходит дальше (если студент уже подтверждал его ранее).
4. GitHub редиректит браузер обратно на `redirect_uri` с параметрами `?code=...&state=...` — `state` возвращается ровно тем же, каким был отправлен на шаге 2.
5. Backend (`GET /join/callback`) проверяет `state` (§3.3), затем **сам, сервер-сервер, а не через браузер студента**, вызывает:
   ```
   POST https://github.com/login/oauth/access_token
   Accept: application/json
   client_id={GITHUB_OAUTH_CLIENT_ID}
   client_secret={GITHUB_OAUTH_CLIENT_SECRET}
   code={code из шага 4}
   redirect_uri={тот же callback URL}
   ```
   В ответе приходит `access_token` — короткоживущий токен студента, нужный нам ровно для одного следующего запроса.
6. Backend вызывает:
   ```
   GET https://api.github.com/user
   Authorization: Bearer {access_token из шага 5}
   ```
   и получает поле `login` — это и есть **подтверждённый GitHub-username**, дальше используемый как `username` в логике §4. Именно этот шаг — единственный источник истины об идентичности студента; username, полученный любым другим путём (например, если бы его вместо этого сообщал frontend), не в счёт.
7. Access token студента после шага 6 нигде больше не используется и не должен сохраняться (ни в логах, ни в Google Таблице, ни где-либо ещё) — он одноразовый и не нужен после того, как получен `username`.

### 3.3 Зачем нужен `state` и что произойдёт, если его не проверять

- `state` — подписанный (не зашифрованный: важно не то, что его нельзя прочитать, а то, что его нельзя подделать) токен, содержащий `course_id`, `lab_id` и время создания.
- Подпись и срок действия (например, 10 минут) проверяются на шаге 5 тем же `itsdangerous`-подписчиком, что уже используется в проекте для `admin_session` (main.py:250/270) — отдельное серверное хранилище сессий не нужно.
- Если подпись/срок `state` не проверять — колбэк можно вызвать напрямую, минуя шаги 1–4, подставив произвольные `course_id`/`lab_id`. Раз эти идентификаторы не секретны, ущерб ограничен (репозиторий в худшем случае создастся «не под той» лабой), но проверка подписи и срока действия всё равно обязательна — это стандартная практика OAuth, а не специфика проекта, и её отсутствие — ошибка, а не мелкая недоработка.
- Отличие от классического «OAuth login CSRF»: здесь `state` не открывает постоянную сессию с правами доступа — он лишь направляет уже полученный от GitHub `code` в правильный контекст (какую лабу обрабатывать). Это снижает серьёзность возможных огрехов в этой части, но не отменяет обязательности проверки при приёмке.

## 4. Логика создания и проверки доступа (эндпоинт колбэка)

Выполняется на каждый успешный OAuth-колбэк (то есть при каждом переходе студента по ссылке, включая повторные):

1. По `state` восстановить `course_id`, `lab_id`.
2. Обменять `code` на access token, вызвать `GET /user` → получить подтверждённый `username`.
3. Загрузить конфиг лабы: `organization`, `github-prefix`, новое поле `template-repo` (см. §6).
4. Вычислить `repo_name = f"{github-prefix}-{username}"`.
5. Проверить существование `{organization}/{repo_name}`:
   - не существует → `POST /repos/{template_owner}/{template_repo}/generate` с `owner=organization, name=repo_name, private=true`;
   - существует → пропустить создание, содержимое репозитория не проверяется и не трогается (сознательное решение — упрощение объёма первой версии).
6. Проверить доступ студента к репозиторию:
   - `GET /repos/{organization}/{repo_name}/collaborators/{username}?affiliation=direct` → 204 → доступ уже есть, ничего не делать;
   - иначе `GET /repos/{organization}/{repo_name}/invitations`, найти приглашение с `invitee.login == username`:
     - найдено (pending) → `DELETE /repos/{organization}/{repo_name}/invitations/{id}`, затем заново `PUT /repos/{organization}/{repo_name}/collaborators/{username}` (повторный `PUT` без удаления старого приглашения не пересылает уведомление — это и есть причина существования github-reinvite, поэтому необходимы оба вызова);
     - не найдено (не приглашался или приглашение истекло за 7 дней) → `PUT /repos/{organization}/{repo_name}/collaborators/{username}` напрямую.
7. Показать студенту страницу успеха со ссылкой на `https://github.com/{organization}/{repo_name}` — не на письмо/уведомление. Поскольку студент только что прошёл OAuth, он гарантированно залогинен в браузере, и при переходе по этой ссылке GitHub сам покажет баннер принятия приглашения на странице репозитория, если оно ещё не принято.

Проверка/починка доступа выполняется только реактивно, по факту захода студента по ссылке — фоновая/периодическая проверка всех репозиториев по расписанию в объём этой фичи не входит (решение принято осознанно, чтобы не вводить в проект планировщик/cron).

## 5. Изменения конфигурации курса

Новое поле в секции `labs.*` (документируется в `docs/COURSE_CONFIG.md` по аналогии с существующими полями):

### `template-repo` (обязательно для использования этой фичи)
**Тип:** `string`
**Описание:** Репозиторий-шаблон в формате `owner/repo`, из которого создаётся репозиторий студента. Репозиторий должен быть отмечен как Template repository в настройках GitHub.
**Пример:**
```yaml
labs:
  "1":
    github-prefix: os-task1
    template-repo: suai-os-2025/os-task1-template
```

Если для лабы фича не используется (нет `template-repo`), ссылка `/join/{course_id}/{lab_id}` должна отдавать понятную ошибку конфигурации, а не падать необработанным исключением.

## 6. Новые переменные окружения

```bash
GITHUB_OAUTH_CLIENT_ID=...
GITHUB_OAUTH_CLIENT_SECRET=...
```

Требование к существующему `GITHUB_TOKEN`: аккаунт, которому он принадлежит, должен иметь право создавать репозитории в организации(-ях) курсов (владелец организации или соответствующая роль). Отдельный GitHub App с более узкими правами вместо classic PAT рассматривается как возможное будущее усиление безопасности, но не входит в объём первой версии.

## 7. Новые эндпоинты backend (main.py)

| Эндпоинт | Назначение |
|---|---|
| `GET /join/{course_id}/{lab_id}` | Публичная информация для лендинга (название курса, короткое имя лабы), без аутентификации |
| `GET /join/{course_id}/{lab_id}/start` | Строит подписанный `state`, редиректит на `github.com/login/oauth/authorize` |
| `GET /join/callback` | Обрабатывает колбэк OAuth: проверяет `state`, получает `username`, выполняет логику §4, редиректит на фронтенд-страницу результата |

Ошибки, которые нужно явно обработать (не как необработанные исключения):
- неизвестные `course_id`/`lab_id`;
- в лабе не настроен `template-repo`;
- студент отклонил доступ на стороне GitHub (`error=access_denied` в колбэке) — понятное сообщение с возможностью повторить;
- невалидный/просроченный `state`;
- ошибки GitHub API при создании репозитория (гонка при одновременном создании с тем же именем, недоступный шаблон, rate limit) — сообщение с возможностью повторить, без падения 500.

Как и остальные публичные эндпоинты проекта, использовать `@limiter.limit(...)` (slowapi уже подключён и используется, см. `read_index`, `admin_login`).

## 8. Новое на frontend

- Компонент `frontend/courses-front/src/components/JoinLab/index.jsx` (+ `styled.js`) по существующей конвенции.

## 9. Не входит в объём (явно)

- Изменения в `/register` и в логике проверки/оценки лабораторных работ (`grade_lab`).
- Флаг «лаба открыта/закрыта» — ссылку можно использовать в любой момент.
- Проверка содержимого уже существующего репозитория.
- Периодическая/фоновая проверка приглашений по расписанию (только реактивно, по заходу на ссылку).
- Групповые/командные репозитории (один репозиторий на одного студента, как и сейчас в конфиге). Задел на будущее: в реализации §4 суффикс имени репозитория (`{github-prefix}-{username}`) стоит называть в коде нейтрально (например, `repo_suffix`/`join_key`), а не жёстко `username`, чтобы позже для групповых лаб это же значение можно было вычислять как имя команды (`{github-prefix}-{team_name}`) без переписывания логики создания репозитория и проверки/починки доступа — сам алгоритм §4 всегда обрабатывает ровно один username за один визит и для команд не меняется, меняется только то, откуда берётся суффикс.
