# Исследование: детекция ИИ-сгенерированного кода

Предварительное исследование возможности автоматической проверки, является ли студенческий код сгенерированным ИИ (LLM). Рассматривается как потенциальное расширение по аналогии с [`docs/PLAGIARISM_DETECTION_PLAN.md`](./PLAGIARISM_DETECTION_PLAN.md), раздел 7 которого явно исключает эту задачу из объёма фичи проверки на плагиат.

## 1. Отличие от задачи проверки на плагиат

Проверка на плагиат сравнивает работу с конкретным вторым артефактом (другой сдачей, шаблоном) — результат можно предъявить как пару текстов с выделенными совпадениями. Детекция ИИ-кода — zero-shot классификация без эталона для сравнения: инструмент выносит вердикт на основе статистических признаков текста, не имея конкретного "оригинала" для сопоставления. Это структурно отличается тем, что:
- вывод — вероятность/вердикт без конкретного артефакта, который можно предъявить студенту;
- надёжность существующих инструментов ниже, чем у зрелых плагиат-детекторов (JPlag/Dolos/compare50);
- задача носит характер гонки вооружений (перефразирование текста уходит от детекции) в ещё большей степени, чем плагиат-детекция.

## 2. Обзор инструментов и подходов (состояние на 2026 год)

| Подход | Примеры | Точность / надёжность | Инфраструктурные требования |
|---|---|---|---|
| Коммерческие API (текст) | GPTZero, Copyleaks, Originality.ai | Заявлено 98–99%; по независимым тестам false positive rate 10–40%, для не-носителей английского языка — до 61% в отдельных исследованиях | Внешний сервис, отправка кода за пределы инфраструктуры |
| Коммерческие API (код) | Pangram Labs (заявлено 96.2% / FPR 0.3%), Copyleaks Code AI Detector, Codequiry | Маркетинговые показатели выше, чем у текстовых детекторов; независимой валидации на учебном коде (а не на бенчмарках вендора) не найдено | Внешний сервис |
| Zero-shot статистические детекторы | Binoculars (открытый код, [ahans30/Binoculars](https://github.com/ahans30/Binoculars)), DetectGPT, Fast-DetectGPT | Binoculars — TPR>90%, FPR≈0.01% в независимых сравнениях; валидация проводилась на естественном языке (эссе, тексты), не на исходном коде | Требует локального инференса двух LLM (observer/performer) — GPU или медленный CPU-инференс |
| ML-классификаторы под код | GPTSniffer (CodeBERT, [MDEGroup/GPTSniffer](https://github.com/MDEGroup/GPTSniffer)), LLMSniffer (GraphCodeBERT) | Исследовательские прототипы; датасеты преимущественно на Python/Java, перенос на C/C++ не валидирован | Требует дообучения/валидации модели силами реализующего фичу |
| Форензика по git-истории | Сигнатуры инструментов в коммитах (`Co-authored-by: Claude`, боты `copilot-swe-agent[bot]`, `claude[bot]`), конфиг-файлы инструментов (`.cursorrules`, `CLAUDE.md`, `.aider*`), паттерн "один коммит без итераций" | Высокая точность при срабатывании, но низкий recall: по данным исследования на 180M репозиториев (2606.24429, июнь 2026) поиск по bot-аккаунтам находит только 3.3% фактических случаев использования Claude Code | Не требует ML/GPU — переиспользует данные о коммитах, уже получаемые через `grading/github_client.py` |

Общий вывод по независимым исследованиям 2026 года: ни один из перечисленных инструментов не даёт результата, достаточного для использования как самостоятельное доказательство в делах об академической нечестности — вывод детектора служит поводом для дальнейшего разбирательства, а не вердиктом. Vanderbilt University в 2024 году отключил детектор ИИ в Turnitin из-за избыточных ложных срабатываний.

## 3. Специфичный для проекта риск

Задокументированное смещение детекторов против не-носителей английского языка касается русскоязычных студентов курсов, обслуживаемых проектом: код с кириллическими именами переменных и комментариями статистически более склонен ложно классифицироваться как "сгенерированный ИИ" инструментами, откалиброванными на англоязычных корпусах.

## 4. Отличия от архитектуры плагиат-фичи

Архитектура, описанная в `docs/PLAGIARISM_DETECTION_PLAN.md`, опирается на три свойства, не переносящиеся на задачу ИИ-детекции без изменений:
- **Локальность без GPU**: `compare50`/JPlag работают на CPU и встраиваются как pip/JRE-зависимость. Единственный сопоставимый по надёжности локальный подход к ИИ-детекции (Binoculars) требует инференса двух LLM — GPU-инфраструктура, не вписывающаяся в текущий образ `python:3.12-slim` без существенных изменений.
- **Наличие эталона для сравнения**: у плагиат-флага есть конкретная пара работ для сопоставления преподавателем. У ИИ-детекции такого эталона нет — только число/вероятность.
- **Признанная точность метода**: JPlag/compare50/Dolos — инструменты с многолетней историей использования в академической среде и понятной семантикой результата (доля буквально совпадающих токенов). Ни один из рассмотренных инструментов ИИ-детекции не имеет сопоставимой зрелости по состоянию на 2026 год.

## 5. Рекомендация

Не реализовывать бинарный вердикт или числовую "вероятность ИИ" по аналогии с процентом совпадения в плагиат-фиче — при ложном срабатывании отсутствует артефакт для предъявления и разбора со студентом, а задокументированный уровень ложных срабатываний (в т.ч. системно повышенный для не-носителей английского) делает подобный вердикт трудно защитимым.

Основной контроль — уже существующий процесс устной защиты лабораторной работы: он выявляет неспособность объяснить собственный код надёжнее любого автоматического детектора и не создаёт риска автоматического несправедливого обвинения.

Если требуется автоматизация — ограничиться сигналами git-форензики (наличие AI-конфигов инструментов, `Co-authored-by`/bot-сигнатуры в коммитах, единственный итоговый коммит без промежуточных изменений) в качестве advisory-подсказки к защите, без указания на "ИИ" в самой формулировке (например: "репозиторий содержит `.cursorrules`" или "работа сдана одним коммитом без промежуточных изменений — уточнить у студента ход выполнения при защите"). Технически это переиспользует канал комментариев к ячейке (раздел 4.5 `PLAGIARISM_DETECTION_PLAN.md`) и данные о коммитах, уже получаемые через `grading/github_client.py`, без ML/GPU-инфраструктуры. Ограничение метода: по данным исследования на 180M репозиториев такой подход обнаруживает единицы процентов фактических случаев использования ИИ-инструментов.

Полноценный ML-классификатор (GPTSniffer-подобный) или self-hosted Binoculars не рекомендуется в качестве задачи летней практики — результат по точности на конкретных языках курсов (C/C++, Python) не определён заранее, объём работы соответствует исследовательской, а не инженерной задаче с предсказуемым скоупом.

## 6. Не входит в объём

- Числовой скор/вердикт "вероятность ИИ", отображаемый преподавателю.
- Автоматическое влияние на грейд (`v`/`x`) на основе результата ИИ-детекции.
- Самостоятельное обучение или валидация ML-классификатора кода в рамках летней практики.
